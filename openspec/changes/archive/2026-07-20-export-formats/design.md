# Design: Add glTF/GLB and FBX ASCII Export Formats

## Technical Approach

Add pure-JavaScript serializers `exportGLB()` and `exportFBX()` to `voxelizer/voxio.js` alongside the existing `exportOBJ()`/`exportVox()`. Both reuse `result.greedyFacesList`, `result.palette`, and `result.surfaceMaterials`; apply the same centering/scaling as `exportOBJ`; and preserve per-palette PBR data. UI toggles and batch ZIP wiring follow the existing `wantObj`/`wantVox` pattern in `voxelizer/app.js` and `voxelizer/index.html`. Tests run in the existing Node `vm` harness.

## Architecture Decisions

| Decision | Options | Tradeoffs | Choice |
|---|---|---|---|
| Serializer implementation | A) Three.js `GLTFExporter` + FBX SDK wrapper; B) Custom pure-JS serializers | A adds vendor dependency, requires DOM/Three context, harder to test in Node; B is deterministic, testable in `vm`, no new deps | B — custom serializers |
| Mesh topology | A) Unique indexed vertices with deduplication; B) 4 vertices + 6 indices per face | A saves ~50% vertices but adds hashing cost/complexity; B matches OBJ vertex layout, simpler, fast enough for current budgets | B — 4 verts/6 indices per face |
| Index type | A) Always `UNSIGNED_INT`; B) `UNSIGNED_SHORT` when possible | A simpler; B halves index bytes for small models | B — promote to `UNSIGNED_INT` when face count * 6 > 65535 |
| Coordinate system | A) Match OBJ (center, Y-up); B) glTF default (Y-up no center) | A keeps all formats consistent; B would diverge from existing exports | A — center at `(DX/2, DY/2, DZ/2)` and scale like `exportOBJ` |
| GLB vertex colors | A) `COLOR_0` as `VEC3` `FLOAT`; B) `VEC4` `UNSIGNED_BYTE` normalized | A easier to modulate AO; B smaller but needs alpha channel | A — `VEC3` `FLOAT` for AO/vertex color |
| FBX material model | A) Full PBR `PBR` template; B) Legacy `Material` numeric properties | A not supported by all Blender versions; B imports reliably into Blender 3.6+ | B — legacy `Material` with `P:` properties |

## Data Flow

```
greedyFacesList ──┐
palette ──────────┼──► voxio.js face assembler ──► indexed triangle mesh ──► GLB builder ──► Uint8Array
surfaceMaterials ─┘                                      │
                                                          ▼
                                                   FBX builder ──► string
```

The assembler iterates faces, calls `fixWinding()`, emits 4 positions + 6 indices + material index per face, and optionally emits per-vertex colors modulated by AO.

## Interfaces / Contracts

```javascript
function exportGLB(result, opts) // => Uint8Array
function exportFBX(result, opts) // => string
```

`opts` mirrors `exportOBJ`: `scale`, `useAO`, `aoStrength`, `annotateAO`. Add `vertexColors` (default `false`) to emit `COLOR_0`.

`result` contract:
- `greedyFacesList`: `{ corners: number[4][3], normal: number[3], color: number, ao?: number[4] }[]`
- `palette`: `number[N][3]`
- `surfaceMaterials`: `{ metallic, roughness, emissive }[N]`
- `dims`: `[DX, DY, DZ]`

GLB: one mesh primitive per material, indexed triangles, accessor min/max, one bufferView per attribute, JSON padded to 4 bytes, BIN padded to 4 bytes.

FBX: header, `GlobalSettings` (Y-up, UnitScale 1.0), `Objects` with `Model:Mesh`, `Geometry` (`Vertices`, `PolygonVertexIndex` with negative end markers, `LayerElementMaterial`), and `Material` nodes. Materials use legacy `P:` properties.

## Testing Strategy

| Layer | What to Test | Approach |
|---|---|---|
| Unit | GLB header, chunk types, JSON parse, one mesh, accessor counts | Node `vm` harness; parse generated bytes manually |
| Unit | FBX header, `GlobalSettings` Y-up, `Model`, `Geometry`, `PolygonVertexIndex` negative markers | String assertions on output |
| Unit | Coordinate centering + scaling | 1×1×1 voxel with `scale=2.5`; assert centroid zero and extent |
| Unit | PBR material mapping | Inject non-zero `surfaceMaterials`; assert GLB factors and FBX `P:` lines |
| Integration | UI toggles trigger downloads / ZIP entries | Existing app tests via `collectResultFiles` pattern (manual DOM test if needed) |

## Threat Matrix

N/A — no routing, shell, subprocess, VCS/PR automation, executable-file classification, or process-integration boundary.

## Migration / Rollout

No migration required. New formats are additive; existing `.obj`/`.vox` exports remain unchanged.
